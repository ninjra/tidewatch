% ABSTRACT
% ======================================================================
\begin{abstract}
Autonomous agent systems that manage hundreds to tens of thousands of
concurrent obligations need a prioritization signal richer than a bare
deadline.  Tidewatch is a scoring engine that decomposes obligation
urgency into six multiplicative factors (all bounded away from zero):
exponential time-decay, timing-sensitivity amplification,
deadline-violation amplification with exponential decay, materiality
weighting, logistic completion dampening, and temporally gated
dependency fanout.  These factors are retained in a six-dimensional
component space with deferred scalarization, preserving per-factor
interpretability and Pareto-dominance comparison until a consumer
requests a scalar ranking.  An adaptive rate constant spreads the
pressure curve across the full $[0, 1]$ range at any population size;
incremental rescoring, rank normalization, and zone-capacity limits
maintain discrimination at $N{>}10{,}000$ without degrading
small-$N$ behavior.

We evaluate the core engine with 659 deterministic and property-based
tests ($\Delta < 10^{-10}$ against closed-form reference values under
pinned Python~3.11+ with zero runtime dependencies) and
Monte Carlo scheduling simulations (200 trials, seed${=}42$,
LogNormal durations).  EDF is theoretically optimal for deadline
minimization; Tidewatch does not compete on that metric.  At
$N{=}50$ it matches EDF ($6.7\%$ missed deadlines); at $N{=}200$,
EDF achieves $18.5\%$ vs.\ Tidewatch's $20.1\%$---an $8.6\%$
relative cost of encoding materiality, dependency fanout, and
completion state alongside deadline proximity.  Against FIFO and
random baselines, Tidewatch reduces missed deadlines by
$35$--$47\%$.  A capacity-aware reranking extension modulates
presentation order by operator or system load; it is specified and
implemented but not empirically validated.

The primary contribution is an interpretable, auditable multi-factor
ranking architecture with deferred scalarization, designed as the
prioritization substrate for agent orchestration systems.  No field deployment or user study is
presented; external validity remains an open question.
\end{abstract}

% 1  INTRODUCTION
% ======================================================================
\section{Introduction}
\label{sec:intro}

Most task management systems and autonomous agent planners treat
deadlines as step functions: an obligation transitions from
``upcoming'' to ``overdue'' at a single moment, producing either a
binary flag or at best a coarse urgency tier.  This discards the
continuous accumulation of lateness risk---a task due in 2~days
and a task due in 14~days both register as ``not overdue,'' yet the
appropriate allocation of attention differs by an order of magnitude.
Without a continuous signal, priority sorting degenerates to
tie-breaking by title or insertion order.

The problem compounds in agent orchestration systems where autonomous
sessions manage hundreds to tens of thousands of obligations across
multiple repositories simultaneously.  A dispatching agent cannot
simply sort by deadline: it must weigh a legal brief with 50
downstream dependents against a routine config update due tomorrow.
EDF tells it which is due sooner; it cannot tell it which matters
more.  The prioritization substrate must produce a continuous,
decomposable signal that encodes materiality, dependency structure,
and completion state alongside deadline proximity---and must do so
at population scales where na\"{\i}ve scoring saturates into
indistinguishable clusters.

Tidewatch addresses this with two mechanisms:
\begin{enumerate}[nosep]
  \item A \emph{pressure equation} (Section~\ref{sec:pressure}) that
        maps each obligation to a continuous score $P \in [0, 1]$ from
        six multiplicative factors (all bounded away from zero):
        three temporal signals, two state-dependent modifiers, and one
        structural dependency factor.
  \item A \emph{deferred scalarization architecture}
        (Section~\ref{sec:deferred}) that retains all six factors
        in a six-dimensional component space, deferring the product
        collapse until a consumer requires a scalar ranking.  This
        enables Pareto-dominance comparison and weighted aggregation
        without premature information loss.
\end{enumerate}

An optional capacity-aware reranking extension
(Section~\ref{sec:bandwidth}) adapts presentation order to operator
cognitive state or system load via a three-tier risk policy.  This
mechanism is specified and implemented but \emph{not empirically
validated}; it is not part of the evaluated contribution.

Tidewatch is a prioritization substrate, not a scheduler.  It does
not assign start times, allocate resources, or generate Gantt
charts.  It produces an auditable score per obligation and a rank
order, leaving execution decisions to the human operator or the
autonomous agent consuming the signal.  EDF is theoretically optimal
for deadline minimization~\citep{liu1973scheduling}; Tidewatch does
not compete on that metric.  Its contribution is interpretable,
auditable multi-factor ranking with deferred scalarization, designed
for deployment as the prioritization layer in agent orchestration
systems.

% 2  RELATED WORK
% ======================================================================
\section{Related Work}
\label{sec:related}

\textbf{Real-time and OS scheduling.}
EDF~\citep{liu1973scheduling} is optimal for minimizing missed
deadlines on a single resource with known task durations.
Multi-level feedback queues (MLFQ) and Completely Fair Scheduler
(CFS) handle multi-factor prioritization in production operating
systems but target CPU scheduling, not human obligation management.
Tidewatch does not claim to improve on EDF for deadline
minimization; it encodes additional factors that EDF and MLFQ
ignore (materiality, completion state, physiological capacity).

\textbf{Personal productivity systems.}
David Allen's Getting Things Done (GTD) organizes obligations by
context and next-action but provides no continuous urgency signal.
Mark Forster's autofocus algorithms select tasks by ``felt
resistance''---a subjective heuristic.  Tidewatch replaces these
qualitative methods with a quantitative, auditable pressure score.

\textbf{Project management.}
CPM~\citep{kelley1959critical} identifies the longest dependency
chain but assigns equal implicit priority to all critical-path
tasks.  Tidewatch's temporally gated fanout amplifies urgency in
proportion to downstream dependency count \emph{and} deadline
proximity simultaneously.

\textbf{Prioritization heuristics.}
The Eisenhower matrix classifies tasks on two binary axes.
ICE~\citep{ice2016scoring} and RICE~\citep{rice2018intercom} score
by impact, confidence, effort, and reach.
WSJF~\citep{reinertsen2009flow} divides cost-of-delay by duration.
All are static one-time scores without temporal dynamics, completion
tracking, or deferred scalarization.

\textbf{Multi-criteria decision making.}
Weighted-sum, TOPSIS~\citep{hwang1981topsis},
PROMETHEE~\citep{brans1985promethee}, and
AHP~\citep{saaty1980ahp} combine multiple dimensions into a
scalar.  All use additive or outranking aggregation, allowing a high
score on one dimension to compensate for low urgency on another.
Tidewatch's product collapse prevents this: factors are
multiplicative modulators, not interchangeable dimensions.  The
weighted-sum MCDM baseline in Section~\ref{sec:monte_carlo}
normalizes each component to $[0,1]$ by its algebraic bounds before
applying equal weights.

\textbf{Cognitive workload.}
NASA-TLX~\citep{hart1988development} and
Wickens' Multiple Resource Theory~\citep{wickens2008multiple}
measure or model cognitive load.  Tidewatch's capacity-aware
extension uses similar dimensions to modulate task presentation,
but this module is specified only---no empirical validation is
presented.

\textbf{LLM-based agents.}
AutoGPT~\citep{autogpt2023}, BabyAGI~\citep{babyagi2023},
ReAct~\citep{yao2023react}, Reflexion~\citep{shinn2023reflexion},
and the survey by \citet{wang2024survey} address autonomous task
execution.  \citet{park2023generative} and
\citet{khot2023decomposed} explore memory-augmented and modular
agent architectures.  None incorporate a continuous pressure signal
with deferred scalarization or capacity-aware reranking.

\section{The Tidewatch Model}

% ------------------------------------------------------------------
% 3.1  Pressure Equation
% ------------------------------------------------------------------
\subsection{Pressure Equation}
\label{sec:pressure}

Each obligation carries a deadline, a materiality class, a count of
downstream dependents, a completion percentage, a days-in-status
counter, and a violation count.

\paragraph{Factor selection rationale.}
The six factors were selected to cover the three dimensions that
distinguish obligation urgency from a bare deadline: \emph{temporal}
(how soon, how stagnant, how many past failures),
\emph{state-dependent} (how important, how much is done), and
\emph{structural} (how many others are blocked).  This decomposition
is not claimed to be exhaustive---other factors (estimated effort,
domain complexity, resource availability) could extend the model.
These six were selected from operational experience, not from
systematic search (no forward/backward selection or
information-theoretic criterion was applied).
The ablation study in Section~\ref{sec:ablation_study} quantifies
each factor's marginal contribution.

Given the current time, Tidewatch computes a pressure score as the
product of six factors, all bounded away from zero:

\begin{equation}
\label{eq:pressure}
P = \min\!\bigl(1,\;
  P_{\text{time}} \times M \times A \times D
  \times T_{\text{amp}} \times V_{\text{amp}}\bigr)
\end{equation}

\paragraph{Time pressure $P_{\text{time}}$.}
For $t$ days remaining:
\begin{equation}
\label{eq:time}
P_{\text{time}}(t) =
\begin{cases}
  1 - \exp\!\left(\dfrac{-k}{\max(t, \epsilon)}\right)
    & \text{if } t > 0 \\[6pt]
  1.0 & \text{if } t \leq 0 \;\text{(overdue)}
\end{cases}
\end{equation}
where $k = 3.0$ is the rate constant and $\epsilon = 0.01$ prevents
division by zero.
The curve satisfies three design requirements:
(1)~monotonically increasing as $t \to 0$,
(2)~asymptotically approaching~$1.0$, and
(3)~producing intuitive intermediate values
($P_{\text{time}}(7) \approx 0.35$,
 $P_{\text{time}}(1) \approx 0.95$).

\paragraph{Materiality multiplier $M$.}
\begin{equation}
M =
\begin{cases}
  1.5 & \text{if materiality} = \texttt{material} \\
  1.0 & \text{if materiality} = \texttt{routine}
\end{cases}
\end{equation}

\paragraph{Dependency amplifier $A$ with temporal gating.}
Dependencies are amplified proportionally to fanout, but gated by
deadline proximity so that distant deadlines do not receive spurious
amplification from large dependency counts:
\begin{equation}
\label{eq:dep}
A = 1 + \bigl(n_{\text{deps}} \times \alpha_{\text{dep}}
    \times g(t)\bigr)
\end{equation}
where $\alpha_{\text{dep}} = 0.1$ and the temporal gate
$g(t)$ uses an independent rate constant $k_f$ decoupled from the
base time-pressure $k$:
\begin{equation}
\label{eq:gate}
g(t) =
\begin{cases}
  1 - \exp\!\left(\dfrac{-k_f}{\max(t, \epsilon)}\right)
    & \text{if } t > 0 \\[4pt]
  1.0 & \text{if } t \leq 0
\end{cases}
\end{equation}
with $k_f = 2.0$ (independent of $k = 3.0$).
The lower $k_f$ gives dependency amplification a wider activation
window than base time pressure: dependencies start mattering slightly
later, matching the intuition that cascading failure risk materializes
after the deadline itself becomes urgent.
Effective dependency count is capped at $n_{\text{max}} = 20$ to
prevent artificial saturation from spurious fanout.
At $t = 30$ days, $g(30) \approx 0.064$---an obligation blocking 10
others receives only $A \approx 1.064$.
At $t = 2$ days, $g(2) \approx 0.63$, yielding $A \approx 1.63$ for
the same fanout.

\paragraph{Completion dampener $D$.}
A logistic sigmoid models the non-linear relationship between
completion progress and residual urgency:
\begin{equation}
\label{eq:damp}
D = 1 - \beta \cdot \sigma\!\bigl(\kappa \cdot (c - 0.5)\bigr)
\end{equation}
where $\beta = 0.6$ is the maximum dampening, $\kappa = 8$ controls
steepness, $c \in [0, 1]$ is the completion fraction, and $\sigma$
is the standard logistic function.
At $c = 0$: $D \approx 0.99$ (negligible dampening).
At $c = 0.5$: $D = 0.70$ (inflection point).
At $c = 1.0$: $D \approx 0.41$ (residual pressure for verification).
A linear mode ($D = 1 - \beta c$) is available for consumers that
prefer proportional dampening.

\paragraph{Timing amplifier $T_{\text{amp}}$.}
Obligations that remain in the same status for extended periods
receive continuous amplification via a logistic ramp:
\begin{equation}
T_{\text{amp}}(d) = 1 + \frac{\Delta_{\max}}{1 + \exp\!\bigl(-k_t \cdot (d - d_{\text{mid}})\bigr)}
\end{equation}
where $d$ is days in status, $\Delta_{\max} = 0.2$ is the asymptotic
boost, $k_t = 0.5$ controls steepness, and $d_{\text{mid}} = 7$ days
is the inflection point.  At $d = 0$: $T_{\text{amp}} \approx 1.00$;
at $d = 7$: $T_{\text{amp}} \approx 1.10$;
at $d = 14$: $T_{\text{amp}} \approx 1.19$.
The logistic form replaces the prior step function for consistency
with the continuous-framework thesis.

\paragraph{Violation amplifier $V_{\text{amp}}$.}
Past deadline violations increase future pressure, but decay
exponentially to prevent perverse feedback loops where a missed
deadline permanently amplifies an already-difficult obligation:
\begin{equation}
V_{\text{amp}} = 1 + \min\!\bigl(v \cdot 2^{-d / h} \times 0.05,\; 0.5\bigr)
\end{equation}
where $v$ is the cumulative violation count, $d$ is days in current
status, and $h = 14$ is the decay half-life.
The cap at~$1.5$ prevents runaway amplification.
At $d = 0$: full violation effect.
At $d = 14$: half the effect.
At $d = 28$: one quarter.

\paragraph{Saturation bound.}
The $\min(1, \cdot)$ in Equation~\ref{eq:pressure} is the
equation's defined ceiling, not an editorial clamp.
Pressure is a probability-like quantity bounded by $[0, 1]$.
The lower bound is guaranteed by the non-negativity of all six
factors.

% 3.2  Deferred Scalarization
% ------------------------------------------------------------------
\subsection{Deferred Scalarization}
\label{sec:deferred}

Most scoring systems compute factors, multiply them, and discard the
components.  Tidewatch retains all six factors as named dimensions
in a six-dimensional \texttt{ComponentSpace} with four operations:

\begin{enumerate}[nosep]
  \item \textbf{components} $\to$ \texttt{dict[str, float]}:
        raw factor values, keyed by name.
  \item \textbf{collapsed} $\to$ float:
        the product of all components (default collapse).
  \item \textbf{weighted\_collapse}(weights) $\to$ float:
        a weighted sum for query-adaptive ranking.
  \item \textbf{dominates}(other) $\to$ bool$\mid$None:
        Pareto dominance---True if self $\geq$ other on every
        dimension, None if incomparable.
\end{enumerate}

Scalarization happens \emph{only} when a consumer calls
\texttt{collapsed} (product of all six components) or
\texttt{weighted\_collapse}.  Until that point, all six dimensions
are available for inspection, Pareto comparison, or custom
aggregation.

\paragraph{Pareto-dominance ranking.}
The batch recalculation function accepts a \texttt{pareto=True}
flag.  When set, it iteratively extracts Pareto fronts: the first
front contains obligations not dominated by any other; the second
contains those dominated only by first-front members; and so on.
Within each layer, obligations are sorted by their product collapse.
This produces a ranking that respects multidimensional trade-offs
without premature aggregation.

\paragraph{Backend selection.}
The component space has two implementations:
(1)~a full implementation from the Gravitas library (when available),
supporting normalized Pareto dominance and weighted collapse with
configurable bounds; and
(2)~a lightweight fallback using only the standard library,
providing the same protocol with product-only collapse.
Backend selection is automatic and transparent to consumers.

% 3.3  Zone Classification
% ------------------------------------------------------------------
\subsection{Zone Classification}
\label{sec:zones}

The continuous pressure score maps to four operational zones:
\begin{equation}
\text{zone}(P) =
\begin{cases}
  \textsc{green}  & \text{if } P < 0.30 \\
  \textsc{yellow} & \text{if } 0.30 \leq P < 0.60 \\
  \textsc{orange} & \text{if } 0.60 \leq P < 0.80 \\
  \textsc{red}    & \text{if } P \geq 0.80
\end{cases}
\end{equation}
Zone thresholds are environment-configurable.  The zone system
provides an ordinal interface for consumers that do not need
continuous scores (notification routing, display coloring,
plan-request gating).

% ------------------------------------------------------------------
% 3.4  Factor Ablation
% ------------------------------------------------------------------
\subsection{Factor Ablation}
\label{sec:ablation}

The pressure function accepts an \texttt{ablate} parameter: a
frozenset of component names to disable.  Ablated factors are set
to their multiplicative identity ($1.0$), isolating the marginal
contribution of each remaining factor.  For example:

\begin{lstlisting}[caption={Ablating dependency and timing factors.},
  label=lst:ablate]
result = calculate_pressure(
    obligation,
    ablate=frozenset({"dependency_amp",
                      "timing_amp"}),
)
\end{lstlisting}

This mechanism supports both sensitivity analysis
(Section~\ref{sec:sensitivity}) and diagnostic debugging.  Ablating
all six factors yields $P = 1.0$ (the identity element of
multiplication under saturation), confirming that no hidden additive
terms exist.

% 3.5  Cognitive Bandwidth Modulation
% ------------------------------------------------------------------
\subsection{Cognitive Bandwidth Modulation}
\label{sec:bandwidth}

Pressure scores are objective---they reflect deadline reality
independent of who is reading them.
Bandwidth modulation is subjective: it re-ranks the pressure-sorted
queue based on the operator's current cognitive state \emph{without
altering any pressure score}.

\paragraph{CognitiveContext.}
The operator's state is captured as a \texttt{CognitiveContext}
with nine optional signal channels, each normalized to $[0, 1]$
where $1.0$ = optimal:
\begin{itemize}[nosep]
  \item \texttt{sleep\_quality}, \texttt{hrv\_trend},
        \texttt{pain\_level} --- self-reported or wearable-derived
  \item \texttt{hours\_since\_sleep} --- normalized via
        $\max(0, 1 - \max(0, h - 8) / 8)$
  \item \texttt{medication\_window} --- boolean
  \item \texttt{bandwidth\_score} --- pre-computed composite override
  \item \texttt{violation\_rate}, \texttt{constraint\_pressure},
        \texttt{session\_load} --- system-derived signals
\end{itemize}
The effective bandwidth $b$ is the arithmetic mean of available
signals, clamped to $[0.2, 1.0]$.  The floor of $0.2$ prevents
compromised or misconfigured telemetry from creating a denial-of-service
state for complex work.  If no signals are present, $b = 0.8$
(fail-safe-conservative: assume mild degradation rather than full
capacity).  A pre-computed \texttt{bandwidth\_score} overrides
the average when provided.

\paragraph{Task demand estimation.}
Each obligation maps to a cognitive demand profile with three
dimensions---complexity, novelty, and decision weight---via
domain-specific profiles:

\begin{table}[t]
\centering
\caption{Domain demand profiles.  Material obligations add $+0.2$
complexity and $+0.1$ decision weight.}
\label{tab:demand}
\begin{tabular}{@{}lccc@{}}
\toprule
\textbf{Domain} & \textbf{Complexity} & \textbf{Novelty} &
  \textbf{Decision} \\
\midrule
legal      & 0.8 & 0.6 & 0.9 \\
financial  & 0.8 & 0.5 & 0.8 \\
engineering & 0.5 & 0.5 & 0.4 \\
ops / admin & 0.3 & 0.2 & 0.2 \\
\bottomrule
\end{tabular}
\end{table}

\paragraph{Fit-score reranking.}
The bandwidth-adjusted sort computes a fit score for each obligation:
\begin{equation}
\label{eq:fit}
f = P \times \bigl(1 - \bar{d} \times (1 - b)\bigr)
\end{equation}
where $\bar{d} = (\text{complexity} + \text{novelty} +
\text{decision\_weight}) / 3$ is the mean demand.
At $b = 1.0$: $f = P$ (pure pressure ordering).
At $b = 0.0$: $f = P(1 - \bar{d})$, maximally penalizing
high-demand tasks.

\paragraph{Three-tier risk classification.}
A critical edge case arises when a binding deadline coincides with
severely reduced bandwidth---the \emph{Critical-Low Paradox}.
Rather than a binary hard-floor flag, Tidewatch uses a three-tier
\texttt{RiskTier} enum:

\begin{enumerate}[nosep]
  \item[\textbf{0}] \textsc{never\_demotable}: bypasses bandwidth
    modulation entirely.  Sort key $(2, P)$ ensures these items
    always rank first.  Assigned explicitly or auto-detected for
    domains whose mean cognitive demand exceeds a configurable
    threshold ($\bar{d} \geq 0.7$) within 24~hours of deadline.
    This signal-based detection avoids hardcoding domain names:
    any domain with a sufficiently high demand profile triggers
    protection automatically.
  \item[\textbf{1}] \textsc{demotable\_with\_floor}: can be reranked
    downward but not below a configurable floor position.
  \item[\textbf{2}] \textsc{fully\_demotable}: freely reranked by
    bandwidth.  Default for all obligations.
\end{enumerate}

% 3.6  Speculative Planning

\subsection{Speculative Planning}
\label{sec:planner}

The \texttt{PlanStubGenerator} (aliased as
\texttt{SpeculativePlanner}) is a synchronous template engine that
produces structured prompt stubs for high-pressure obligations:

\begin{enumerate}[nosep]
  \item \textbf{Filter} pressure results to obligations in yellow,
        orange, or red zones (configurable via
        \texttt{min\_zones}).
  \item \textbf{Sort} by pressure descending and take top
        $N = 3$ (configurable).
  \item \textbf{Generate} a \texttt{PlanRequest} containing
        obligation metadata, pressure decomposition, a structured
        prompt template, and zone-derived delivery urgency.
\end{enumerate}

The caller provides the LLM; Tidewatch never invokes one directly.
Plan completion wraps the LLM response in a \texttt{PlanResult}
with full provenance (obligation ID, zone, pressure, timestamp).

% ------------------------------------------------------------------
% 3.7  Triage Queue
% ------------------------------------------------------------------
\subsection{Triage Queue}
\label{sec:triage}

External scanners produce candidate obligations as
\texttt{TriageCandidate} objects.
The \texttt{TriageQueue} stages them for operator review with
deduplication by (title, source, due\_date) tuple, UUID assignment,
and accept/reject conversion to full \texttt{Obligation} objects.
The queue is in-memory and stateless---the caller handles
persistence, keeping Tidewatch free of database dependencies.

\section{Implementation}
\label{sec:implementation}

Tidewatch is implemented in Python~3.11+ using only standard-library
modules (\texttt{math}, \texttt{datetime}, \texttt{dataclasses},
\texttt{enum}, \texttt{uuid})---no third-party packages are
required at runtime.
The core library comprises 2{,}378 lines across seven modules
(\texttt{pressure}, \texttt{components}, \texttt{types},
\texttt{constants}, \texttt{planner}, \texttt{triage},
\texttt{\_\_init\_\_}).

The pressure engine is explicitly deterministic: no LLM inference,
no database access, no asynchronous I/O, no randomness.  Every
pressure score is reproducible given the same (obligation, now)
inputs.

Input validation uses fail-fast \texttt{ValueError} exceptions:
completion percentage must lie in $[0, 1]$ and dependency count
must be non-negative.  Invalid inputs are rejected at the function
boundary, never silently clamped.

Optional integrations (Sentinel telemetry, Gravitas component
space) degrade gracefully via try/except import guards.  The core
library functions identically with or without these packages.

\subsection{Monte Carlo Simulation}
\label{sec:monte_carlo}

We evaluate scheduling outcomes via a Monte Carlo simulation
comparing thirteen strategies across 200 trials (seed~$= 42$).

\paragraph{Simulation model.}
$N$ obligations with deadlines, dependencies, and
domain-specific processing durations sampled from LogNormal
distributions (e.g., legal: $\mu = 2.0$, $\sigma = 0.8$,
median~$\approx 7.4$~hours).
A single operator resource processes obligations sequentially in the
order determined by the scheduling strategy.

\paragraph{Strategies.}
\begin{enumerate}[nosep]
  \item \textbf{Tidewatch:} order by pressure descending (product collapse).
  \item \textbf{Tidewatch+BW ($b{=}0.5$, $0.2$):} pressure with
        bandwidth-adjusted reranking at fixed medium/low capacity.
  \item \textbf{Tidewatch+BW (variable):} bandwidth sampled from
        $\text{Beta}(2, 3)$ per trial (mean $\approx 0.4$).
  \item \textbf{Weighted-sum:} MCDM baseline---normalized
        weighted-sum collapse with equal weights, each component
        scaled to $[0, 1]$ by its algebraic bounds.
  \item \textbf{TOPSIS:} MCDM baseline with domain-weighted
        profiles (time${}=0.3$, materiality${}=0.2$,
        dependency${}=0.2$, completion${}=0.15$,
        timing${}=0.1$, violation${}=0.05$).
  \item \textbf{Weighted-EDF:} hybrid---zone-tier urgency
        (red$>$orange$>$yellow$>$green) with EDF within each tier.
  \item \textbf{LLF:} least laxity first (deadline minus estimated
        remaining duration).
  \item \textbf{Slack:} remaining-slack order (deadline minus
        remaining work, accounting for partial completion).
  \item \textbf{EDF:} earliest deadline first.
  \item \textbf{FIFO:} insertion order.
  \item \textbf{Random:} shuffled (null hypothesis).
\end{enumerate}

\paragraph{Outcome metrics.}
\begin{itemize}[nosep]
  \item \emph{Missed-deadline rate}: fraction of obligations
        completed after their deadline.
  \item \emph{Queue-inversion rate}: fraction of processing steps
        where a lower-pressure item is processed while a
        higher-pressure one waits.
  \item \emph{Attention efficiency}: fraction of time spent on
        obligations that ultimately met their deadline.
\end{itemize}

\paragraph{Key findings.}
Queue inversions are measured by recalculating pressures at the
current simulation clock at each processing step---not from a stale
snapshot at simulation start.  As obligations are processed, remaining
deadlines approach, causing pressure to shift.  Tidewatch's
$1.9$--$4.0\%$ inversion rate reflects these live-clock pressure
changes: the initial ordering is correct at $t{=}0$, but processing
time causes the ranking to drift.  EDF's $6.5$--$11.8\%$ inversion
rate is measured on the same live-clock pressure metric.

On missed-deadline rate---an exogenous metric independent of any
strategy's internal ordering---Tidewatch matches EDF at $N{=}50$ and
trails by~$9\%$ at $N{=}200$ ($20.1\%$ vs.\ $18.5\%$).  This is
expected: EDF is theoretically optimal for single-resource,
deadline-only scheduling.  Tidewatch's additional factors
(materiality, dependencies, completion) can cause it to prioritize
non-deadline-urgent but otherwise important tasks, occasionally
displacing a barely-deadline-urgent item.

Bandwidth-adjusted variants improve missed-deadline rates at
$N{=}200$: TW+BW ($b{=}0.5$) achieves $20.0\%$ and TW+BW ($b{=}0.2$)
achieves $19.5\%$, narrowing the gap with EDF ($18.5\%$).  This
suggests that bandwidth reranking surfaces tasks that are both
deadline-safe and execution-feasible.

Against na\"{\i}ve baselines (FIFO, Random), Tidewatch reduces
missed deadlines by $35$--$47\%$ and improves attention efficiency
by $3$--$4\%$.

\paragraph{Saturation and tie-breaking.}
The $\min(1, \cdot)$ bound means multiple obligations can saturate at
$P = 1.0$ simultaneously.  When ties occur, obligations are ordered by
their position in the input list (stable sort).  This is a known
information-loss boundary: the deferred-scalarization component space
preserves the raw (unclamped) product for consumers that need
discrimination above the ceiling, but the default product collapse
does not.  When this occurs, the implementation uses the raw
(unclamped) product as a secondary sort key, resolving ties by
actual urgency rather than insertion order.
Under adversarial conditions (all $N{=}50$ obligations due within
24~hours), the saturation rate reaches $76\%$ and $57\%$ of
ranking decisions are tie-affected.  Both Tidewatch ($98\%$
missed) and EDF ($99.7\%$ missed) fail catastrophically in this
regime; the system is not designed for burst-deadline scenarios.

\paragraph{Bandwidth reranking.}
At $b = 0.5$ (medium capacity), bandwidth-adjusted reranking
\emph{improves} deadline compliance ($0.060$ vs.\ $0.067$ at
$N{=}50$; $0.200$ vs.\ $0.201$ at $N{=}200$).
At $b = 0.2$ (low capacity), the same pattern holds with further
improvement at $N{=}200$ ($0.195$).
This suggests that bandwidth reranking surfaces easier tasks \emph{that
are still deadline-safe}, improving effective throughput.

\paragraph{Product collapse vs.\ weighted-sum MCDM.}
The weighted-sum baseline (equal weights across six factors) produces
$73\%$ more missed deadlines than product collapse ($0.116$ vs.\
$0.067$ at $N{=}50$) and substantially more queue inversions
($0.315$ vs.\ $0.019$).
This occurs because additive aggregation treats all factors as
interchangeable, allowing a high score on one dimension (e.g.,
materiality) to compensate for low urgency---a property that
product collapse avoids by design.

TOPSIS with domain-weighted profiles performs \emph{worse} than
naïve weighted-sum at $N{=}200$ ($0.328$ vs.\ $0.320$).  This
occurs because TOPSIS's distance-from-ideal calculation amplifies
the dominance of time pressure (weight $0.3$, highest single
factor), causing obligations with moderate urgency across
multiple dimensions to rank above deadline-critical items.
The result confirms that product collapse---where time pressure
is a multiplicative base signal, not an additive
dimension---better preserves deadline-relevant ordering.

LLF and Slack baselines converge with EDF at both scales
(${\sim}18.6\%$ at $N{=}200$).  Under single-resource sequential
processing, laxity and remaining-slack ordering are near-equivalent
to deadline ordering because the operator processes one obligation
at a time---the duration estimate matters only when it determines
whether slack is sufficient, not when it affects parallel
scheduling decisions.

\subsection{Sensitivity Analysis}
\label{sec:sensitivity}

The rate constant $k$ is the primary tunable parameter.
The design constraint is that a 7-day deadline with no other
factors should enter yellow ($P \geq 0.30$):

\begin{itemize}[nosep]
  \item $k = 2$: yellow entry at ${\sim}5$ days (too late)
  \item $k = 3$: yellow entry at ${\sim}7$ days (chosen value)
  \item $k = 4$: yellow entry at ${\sim}10$ days (premature)
\end{itemize}

The $k = 3$ curve additionally satisfies:
(1)~14-day deadline stays green ($P_{\text{time}}(14) = 0.19$),
avoiding alert fatigue, and
(2)~3-day deadline enters orange ($P_{\text{time}}(3) = 0.63$),
triggering active attention.

``interpretable, auditable'' contribution claim: because each
factor has an independently measurable effect on scheduling
outcomes, a consumer can diagnose any ranking decision by
comparing ablated and unablated scores.  In a multiplicative model,
this attribution is faithful by construction: removing a factor
literally removes its causal contribution.  Factor-level attribution
---``this task ranks higher because its dependency fanout increased
its pressure by $X\%$''---is a direct consequence of the
multiplicative architecture with named, inspectable components.

Bootstrap 95\% CIs (10{,}000 resamples of trial-level outcomes,

\section{Discussion}
\label{sec:discussion}

\paragraph{The EDF gap is the cost of multi-factor ranking.}
Tidewatch does not beat EDF on raw deadline compliance---by design.
EDF orders by deadline alone; Tidewatch encodes five additional
dimensions.  The $8.6\%$ relative cost at $N{=}200$ ($20.1\%$ vs.\
$18.5\%$ missed) is the price of being able to distinguish a legal
brief with 50 downstream dependents from a routine config update
due on the same day.  EDF cannot make that distinction.  For an
agent orchestration system dispatching work across repositories,
this discrimination is the primary requirement---deadline compliance
alone is necessary but insufficient.  Bandwidth-adjusted variants
narrow the gap further: TW+BW ($b{=}0.2$) achieves $19.5\%$,
within $1.0$~pp of EDF.  When deadline compliance is the sole
objective and no multi-factor signal is needed, EDF remains the
correct choice.

\paragraph{Deferred scalarization as an architectural pattern.}
Deferred scalarization is an established technique in MCDM
pipelines~\citep{hwang1981topsis}.  Tidewatch's contribution is
not the technique itself but its application to obligation
prioritization with a specific six-factor decomposition grounded in
deadline mechanics.  In an agent orchestration system, different
consumers need different views of the same obligation: a dispatching
agent needs a scalar rank, a monitoring dashboard needs zone
classification, and a diagnostic tool needs per-factor decomposition.
Deferred scalarization serves all three from a single computation.
The broader pattern---retaining scored dimensions until the consumer
requests a scalar---has applicability beyond obligation
prioritization in any system combining multiple factors (risk
assessment, search ranking, recommendation).

\paragraph{Bandwidth under simulation.}
The Monte Carlo results (Tables~\ref{tab:mc50} and~\ref{tab:mc200})
include bandwidth-adjusted variants that modify sorting order using
a fixed bandwidth parameter.  These variants are included to
characterize the \emph{sorting behavior} of the bandwidth mechanism,
not to validate its efficacy as a cognitive adaptation tool.
The bandwidth module is specified and implemented but not
empirically validated against real physiological data; the
simulation results show only that the reranking mechanism
produces different (and sometimes deadline-favorable) orderings
when bandwidth is reduced.
Intra-trial capacity dynamics and real-signal validation remain
future work.

\paragraph{Deployment safety for physiological reranking.}
The cognitive bandwidth modulator raises deployment considerations
beyond model accuracy:
\begin{itemize}[nosep]
  \item \textbf{Consent:} physiological signals (pain, sleep, HRV) are
    sensitive health data.  Any deployment must obtain explicit opt-in
    and support full opt-out without degrading the base prioritization.
  \item \textbf{Override precedence:} the operator must be able to
    manually override any bandwidth-driven demotion.  The three-tier
    risk classification provides structural override via
    \textsc{never\_demotable}, but per-obligation manual override
    is not yet implemented.
  \item \textbf{Audit logging:} every bandwidth-driven reranking should
    produce an auditable record of which signals were used, what the
    effective bandwidth was, and which obligations were reranked.
    The current implementation logs at debug level; production use
    requires persistent audit trails.
  \item \textbf{Failure modes:} if signal quality degrades (stale
    wearable data, sensor malfunction), the system defaults to
    $b = 0.8$ (conservative) rather than $b = 1.0$ (optimistic).
    A bandwidth floor of $0.2$ prevents poisoned signals from
    suppressing all high-demand work.
  \item \textbf{Gaming:} self-reported signals are gameable.
    Completion percentage and dependency count directly affect pressure
    scores and can be manipulated to suppress urgency.  The
    implementation provides provenance fields (source attribution,
    update timestamps) and logs warnings when high-impact inputs lack
    audit trails.  Integration with authenticated wearable APIs and
    system-of-record provenance would further mitigate gaming risks.
\end{itemize}
This paper does not claim production readiness for physiological
reranking.  The mechanism is presented as a research contribution
with explicit safety boundaries.

\paragraph{Limitations.}
\begin{enumerate}[nosep]
  \item \textbf{Single-operator calibration.}
        Constants ($k = 3.0$, $M = 1.5$, $\beta = 0.6$) were
        tuned for one operator's workflow.  Multi-user deployment
        requires per-operator calibration.
  \item \textbf{Heuristic demand estimation.}
        Task demand is estimated from domain and materiality.
        Richer features (historical completion time, code
        complexity metrics) would improve accuracy.
  \item \textbf{Factor independence assumption.}
        The multiplicative model implicitly assumes that factors
        contribute independently.  In practice, completion and
        timing interact (a task at 90\% completion for 14 days
        is qualitatively different from one at 10\% for 1 day),
        and dependency urgency depends on deadline proximity.
        These interactions are partially captured by the temporal
        gate on dependency amplification, but no formal interaction
        model is provided.
  \item \textbf{No feedback loop.}
        Constants are fixed.  A closed-loop system would observe
        completion patterns and adjust parameters over time.
  \item \textbf{Gravity tiebreak (future work).}
        The implementation includes an optional gravity tiebreak
        score from the Gravitas coherence system, but it is not
        part of the core pressure equation and is not evaluated
        in this paper.  Integration remains future work.
\end{enumerate}

\paragraph{Future work.}
Three directions are planned:
(1)~Physiological integration via wearable sensor streams,
replacing self-reported signals;
(2)~learned constants via outcome-driven optimization; and
(3)~Monte Carlo evaluation of bandwidth modulation under
simulated capacity variation.

\section{Conclusion}
\label{sec:conclusion}

Tidewatch replaces the binary overdue flag with a continuous,
decomposable pressure score from six multiplicative factors
(all bounded away from zero), retained in a six-dimensional
component space with deferred scalarization.  EDF remains optimal
for deadline minimization; Tidewatch's contribution is orthogonal:
an interpretable, auditable prioritization substrate that encodes
materiality, dependency structure, and completion state alongside
deadline proximity.  Adaptive rate-constant selection, incremental
rescoring, rank normalization, and zone-capacity limits extend the
architecture to populations of tens of thousands without degrading
small-$N$ behavior.

The framework has zero runtime dependencies and is verified to
$\Delta < 10^{-10}$ against analytically computed reference values
across 659 deterministic tests under pinned Python~3.11+.  It is
designed for deployment as the prioritization layer in agent
orchestration systems, where a dispatching agent consumes the
pressure signal to decide which obligation to work next.  No field
deployment or user study is presented; external validity remains an
open question.
